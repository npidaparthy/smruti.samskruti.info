data/bg/content/chapters/ch01.json … ch18.json ✓ EDIT data

One file per chapter. Loaded by the app at runtime. Contains verse text in all 3 scripts, meanings in 3 languages × 3 depths, and the chapter conclusion colophon. This is the primary file to edit for verse corrections.

Top-level structure

{
  "chapter": 1,
  "title": { "en": "...", "te": "...", "sa": "...", "ro": "..." },
  "shlokas": [ ... ],         // array of shloka objects
  "conclusion": { ... }       // chapter-end colophon — added manually
}

Shloka object

{
  "c": 1,              // chapter number (integer)
  "s": 1,              // shloka number (integer)
  "speaker": "dhritarashtra",
  "p1": { "ro": "dharma-kṣetre kuru-kṣetre...", "te": "ధర్మక్షేత్రే...", "sa": "धर्मक्षेत्रे..." },
  "p2": { "ro": "...", "te": "...", "sa": "..." },
  "p3": { "ro": "...", "te": "...", "sa": "..." },
  "p4": { "ro": "...", "te": "...", "sa": "..." },
  "meaning": {
    "en": { "short": "...", "long": "...", "wbw": [ { "word": "dharma", "grammar": "nom.", "meaning": "righteousness" }, ... ] },
    "te": { "short": "...", "long": "..." },
    "sa": { "short": "...", "long": "..." }
  }
}

The 3 script fields

FieldScriptUsed for
roIAST Roman (e.g. dharma-kṣetre)Search index, name-matching, audit script — names matched here only
teTelugu (ధర్మక్షేత్రే)Reader display in Telugu mode
saDevanāgarī (धर्मक्षेत्रे)Reader display in Sanskrit mode
All 3 scripts must be updated together when correcting verse text. The app switches between ro/te/dn based on the user's Lipi setting. A fix in only te will still show the wrong text in IAST and Devanāgarī modes.

What is p1/p2/p3/p4?

Each shloka is split into padas (quarter-verses). Standard anuṣṭubh metre = 4 padas × 8 syllables = 32 syllables. The Reader renders each pada on its own line. Some shlokas in ch11 use triṣṭubh metre (11 syllables/pada); these may have unusual pada splits but still use p1–p4. A 5-pada shloka (like BG 13.0, the Arjuna question) adds a p5 field.

If a compound spans two padas (sandhi boundary), the earlier pada ends with "cont": true — the Reader appends a hyphen.

Conclusion colophon (chapter-level)

{
  "conclusion": {
    "ro": "oṃ tatsaditi śrīmadbhagavadgītāsūpaniṣatsu ... sāṃkhyayogo nāma dvitīyo'dhyāyaḥ",
    "te": "ఓం తత్సదితి ...",
    "sa": "ॐ तत्सदिति ...",
    "meaning": {
      "en": { "short": "Thus ends the second chapter..." },
      "te": { "short": "ఇట్లు శ్రీమద్భగవద్గీతలో..." },
      "sa": { "short": "एवं श्रीमद्भगवद्गीतासूपनिषत्सु..." }
    }
  }
}

Ch18 uses a longer form including the Mahābhārata lineage (śrīmanmahābhārate, śatasāhasrikāyāṃ …) and ends with śrīkṛṣṇārpaṇamastu. The Reader shows this below the last verse of each chapter.

How to fix a verse text error

1
Open data/bg/content/chapters/chXX.json in your editor. Search for "s": 37 to jump to shloka 37.
2
Locate the pada with the error (p1/p2/p3/p4). Fix the text in all 3 script fields: ro, te, dn.
3
Save. Hard-refresh the browser (⌘⇧R) to bypass the service worker cache.
4
If this verse is referenced in data/bg/content/bg-meta.json occurrence lists, check if that also needs updating.
⛔ Do NOT run build.py after directly editing ch*.json files.
build.py regenerates all chapter files from an external source. Running it after manual edits will overwrite everything you changed. Only run build.py if you are rebuilding from the original source data (rarely needed).

✓ DO

  • Edit verse text directly in ch*.json
  • Update all 3 scripts (ro/te/dn) together
  • Hard-refresh (⌘⇧R) to test changes
  • Keep "c" and "s" values as integers, not strings
  • Keep padas in order: p1 → p2 → p3 → p4
  • Commit chapter files separately from meaning files

✗ DO NOT

  • Run build.py after manual edits
  • Change shloka numbers ("s") — they must match bg-meta.json refs
  • Add or remove shlokas without updating bg-meta.json occurrences
  • Edit gita-index.json directly (it is generated)
  • Leave a meaning field empty — use "" rather than deleting the key

To update chapter names — edit the source, not this file

1
Edit data/bg/source/chapter-titles.txt — this is the single source of truth for chapter names in all 4 forms (IAST, Telugu, English, Sanskrit).
2
Run python3 scripts/build.py — this regenerates data/bg/content/gita-index.json. Only run this when no manual edits to ch*.json are pending.
3
Verify: open the app and check the chapter label in the Reader.
data/bg/content/bg-meta.json ✓ EDIT data

Stores metadata about the Gita: Krishna and Arjuna name occurrences used by the Names Review overlay, plus chapter themes, chapters metadata. The name section is the part that needs ongoing curation.

Structure

{
  "names": {
    "krishna": [ ... ],   // 14 names of Krishna
    "arjuna":  [ ... ]    // 16 names of Arjuna
  },
  "chapters": [ ... ],    // chapter theme info — do not touch
  ...
}

Each name entry

{
  "name":       "Kṛṣṇa",       // display name
  "iast":       "Kṛṣṇa",       // REQUIRED — used for name matching in audit
  "te":         "కృష్ణ",         // Telugu display
  "sa":         "कृष्ण",         // Devanāgarī display
  "meaning_en": "The dark one",
  "meaning_te": "...",
  "count":      16,             // MUST equal occurrences.length
  "occurrences": [ "1.14", "1.32", "2.4", ... ]   // "ch.shloka" strings, sorted
}

Rules for occurrences[]

RuleDetail
String format"ch.shloka" — e.g. "1.4", "18.78". Never zero-padded ("01.04") or numeric (18.78).
Sorted numericallyAscending by chapter then shloka: 1.1 → 1.47 → 2.1 → … → 18.78
count must match"count" must always equal occurrences.length. Update both together or the Names Review UI will show wrong counts.
No duplicatesEach ref appears at most once per name.
Verify before addingOpen the chapter JSON, find the shloka, confirm the name (or a sandhi form) actually appears in the ro text.
Name matching uses ro text only. The audit script normalises both the name's iast field and the verse's ro padas. Names that appear only in te or dn text are invisible to the matcher. This is why some names (Arjuna, Anagha) show many false negatives — the ro field may use a different sandhi form.

✓ DO

  • Always keep count == occurrences.length
  • Use no-zero-padding: "1.4" not "01.04"
  • Keep occurrences[] numerically sorted
  • Verify in ch*.json before adding/removing a ref
  • Run python3 scripts/audit-names.py after editing to check

✗ DO NOT

  • Use numbers instead of strings in occurrences
  • Edit iast or sa — they drive matching logic
  • Leave count out of sync with the array length
  • Edit the chapters or themes sections without good reason

1
Edit data/vsn/source/verses/vsn-4-padas.txt (Devanāgarī), vsn-4-padas_iast.txt, and/or vsn-4-padas_te.txt. Keep all 3 in sync — the build reads each directly, no programmatic transliteration is done for shlokas.

Run python3 scripts/build-vsn.py.
3
Verify output in data/vsn/content/vsn-shlokas.json — check p1–p4 split across all 3 scripts.

⚠️ External Repository Dependencies

One build script still depends on a file in another repo on your machine.

Gita verse source — used by build.py

External file: ~/Projects/claude/karthikeya/assets/gita-data-all-padas.json

Contains the original 4-pada verse data for all 700 Gita shlokas. build.py checks data/bg/source/gita-data-all-padas.json first; if not found, falls back to the karthikeya path.

To remove this dependency: copy the file once into this repo:

cp ~/Projects/claude/karthikeya/assets/gita-data-all-padas.json \
   ~/Projects/claude/smruti.samskruti.info/data/bg/source/gita-data-all-padas.json

After that, build.py will use the local copy and no longer needs karthikeya. Add it to .gitignore if you don't want it committed.

VSN dependency resolved ✓ — build-vsn.py previously read an external vsn-4-padas.txt. The source files are now inside the repo at data/vsn/source/ (DN + IAST + Telugu, all 3 scripts). No external dependency remains for VSN.
