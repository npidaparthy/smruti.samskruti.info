BG Data Files — Reference Guide

Use this page while manually editing data/bg-meta.json. Covers every data file, its purpose, structure, and editing rules.

1. Chapter files — data/chapters/ch01.json … ch18.json

data/chapters/ch01.json ✓ EDIT for verse fixes ⛔ NOT for name data

One file per chapter. The app loads these at runtime to render the Reader, search, and names-review. Verse text corrections go here. Name occurrence data lives only in bg-meta.json.

Top-level structure

{
  "chapter": 1,
  "shlokas": [
    {
      "c": 1,    // chapter number (int)
      "s": 1,    // shloka number (int)
      "p1": { "ro": "dharma-kshetre kuru-kshetre ...", "te": "ధర్మ క్షేత్రే కురు క్షేత్రే..." },
      "p2": { "ro": "mama putras tava caiva ...",      "te": "మామ పుత్రాః తవ చైవ..." },
      "p3": { "ro": "...", "te": "..." },   // anushtup shlokas have all 4 padas
      "p4": { "ro": "...", "te": "..." }    // tristubh/other metres may also have 4
    },
    ...
  ]
}

What are p1 / p2 / p3 / p4?

Each shloka is split into padas (quarter-verses). Most BG shlokas use the anushtubh metre — 4 padas of 8 syllables each, giving 32 syllables total. Each pada object has two fields:

FieldMeaningUsed by
p1–p4Pada 1–4 (quarter-verse)Reader renders each on its own line
roRomanised IAST textSearch index, name-matching, names-review audit
teTelugu script textReader display (Telugu mode), search also scans this
Name matching searches ro text only. If a name appears only in te (Telugu script) the audit script will not find it. This is why Arjuna / Anagha show as "wrong" for several verses — the ro field may be missing or use a different transliteration. Check both ro and te when verifying.

To fix a verse text error

StepWhatDetail
1. Finddata/chapters/chXX.jsonOpen the right chapter file. In VS Code search for "s": 37 to jump to shloka 37.
2. Editro or te field of the right padaFix the text directly. Keep IAST diacritics correct in ro; keep Telugu Unicode correct in te.
3. Save—App picks it up on next page reload (service worker may cache — hard-refresh if needed).
Do not run build.py after editing chapter files directly. The build script regenerates all ch*.json from an external source file (karthikeya/assets/gita-data-all-padas.json) and will overwrite your manual fixes. Only run build.py if you are rebuilding from source.

How the app uses chapter files

  • Reader fetches ch{N}.json on demand when user navigates to a chapter
  • Search fetches all 18 on first search to build an in-memory index
  • Names-review overlay loads all 18 to run the occurrence audit
  • Verse reference format: "ch.shloka" string — e.g. "18.78", "1.4"

3. data/bg-meta.json ★ — The file you are fixing

data/bg-meta.json META

Stores metadata about the Gita: name lists for Krishna and Arjuna, chapter themes, and other cross-chapter info. The names section is what needs fixing.

Top-level structure

{
  "names": {
    "krishna": [ ... ],   // 14 names
    "arjuna":  [ ... ]    // 16 names
  },
  "themes": { ... },      // chapter themes — do not touch
  ...
}

Each name entry

{
  "name":        "Kṛṣṇa",          // display name (may differ from iast)
  "iast":        "Kṛṣṇa",          // REQUIRED for matching — use this for search
  "te":          "కృష్ణ",           // Telugu script
  "sa":          "कृष्ण",           // Sanskrit (Devanagari) — do not edit
  "meaning_en":  "The dark one",
  "meaning_te":  "...",
  "count":       16,               // MUST equal occurrences.length
  "occurrences": [                 // list of "ch.shloka" strings
    "1.14", "1.32", "2.4", ...
  ]
}

Rules for occurrences[]

RuleDetail
FormatAlways "ch.shloka" as a string — e.g. "18.78", "1.4". Never "01.04" or 18.78 (number).
SortedNumerically ascending by chapter then shloka. 1.1 → 1.47 → 2.1 → … → 18.78.
No duplicatesEach ref appears at most once per name.
count must match"count" field must always equal the length of "occurrences". Update both together.

6. source/ folder

source/ INPUT ONLY

This folder contains the raw input files used to generate the data/ files. It is NOT loaded by the web app at runtime — it is only used by build/import scripts.

  • Contains original verse text in various formats (CSV, TSV, plain text)
  • Build scripts read from source/ and write to data/
  • If you update data/bg-meta.json directly, the source/ files remain stale — that is fine, they are not served to users
  • You do not need to edit source/ when fixing names manually

7. Do's and Don'ts for manual editing

✓ DO

  • Always keep count == occurrences.length
  • Use format "ch.shloka" (no zero-padding) e.g. "1.4", "18.78"
  • Keep occurrences[] numerically sorted
  • Verify in the chapter JSON before adding or removing a ref
  • Use the audit script after editing to confirm the fix worked
  • Save with UTF-8 encoding (Telugu characters must be preserved)
  • Run python3 scripts/audit-names.py when done

✗ DO NOT

  • Add zero-padding: never "01.04" — always "1.4"
  • Use numbers instead of strings: never 18.78 — always "18.78"
  • Edit iast or sa fields — they affect matching logic
  • Edit name, te, meaning_en/te without good reason
  • Edit chapter files (data/chapters/ch*.json)
  • Edit the themes or other sections of bg-meta.json
  • Leave count out of sync with occurrences.length

8. How to fix names manually — step by step

Workflow

  1. Open data/bg-meta.json in your editor
  2. For each suspicious entry, open the relevant data/chapters/chXX.json
  3. Find the shloka by "s": N and look at the ro text of all padas
  4. Check visually if the name (or a close form) appears in the romanised text
  5. Update occurrences[] and then update count to match

Finding a verse quickly

In VS Code: open the chapter file and search for "s": 78 to jump to shloka 78. Or search by partial text.

Important: Bharata judgement calls

When you see "Bharata" in a verse, decide: is it a vocative address to Arjuna (like "O Bharata") or is it a generic reference to the Bharata lineage? Only the former should be in Arjuna's occurrences list.

  • Vocative address → ADD to Arjuna's occurrences
  • Generic / third-person → leave out
